RB-001
Entra Connect Sync Failure
Delta or full synchronization between Active Directory and Entra ID has stopped, stalled, or is producing export errors. Users, groups, or attribute changes are not propagating to the cloud directory.
P1 — High Hybrid Identity ⏱ Est. 15–45 min
Symptoms & Triggers
⚠️ New AD users not appearing in Entra portal after 30+ minutes
⚠️ Entra Connect Health alert email: "Synchronization service not running"
⚠️ Attribute changes (title, department, group membership) stuck in AD
⚠️ Synchronization Service Manager shows red export error icons
Diagnostic & Resolution Steps
✓
01 Confirm the sync engine service is running ›

Open Services on the Entra Connect server and check the status of Microsoft Azure AD Sync. If the service is stopped, start it manually. If it fails to start, check the Windows Application event log for service crash errors.

PowerShell Get-Service "ADSync" | Select-Object Name, Status, StartType

# If stopped:
Start-Service "ADSync"
ℹ️ The ADSync service runs as a Windows service under the AAD_xxxx account that was created during installation. If the service account password was changed in AD without updating the service, it will fail to start.
✓
02 Check the sync scheduler status ›

The scheduler controls the automatic 30-minute delta sync cycle. Verify it's enabled and not suspended. SyncCycleEnabled must be True and StagingModeEnabled must be False for production sync.

PowerShell 5.1 Import-Module "C:\Program Files\Microsoft Azure AD Sync\Bin\ADSync\ADSync.psd1"
Get-ADSyncScheduler

# Key properties to check:
# SyncCycleEnabled : True
# SyncCycleInProgress : False
# StagingModeEnabled : False
# NextSyncCyclePolicyType : Delta
Is SyncCycleEnabled = False?
YES → Run Set-ADSyncScheduler -SyncCycleEnabled $true then proceed to Step 3
NO → Scheduler is enabled, proceed to Step 3 to check for export errors
✓
03 Open Synchronization Service Manager and inspect run history ›

Launch Synchronization Service Manager from Start Menu (or C:\Program Files\Microsoft Azure AD Sync\UIShell\miisclient.exe). Click the Operations tab. Look at the most recent runs — each row shows a connector name, operation type (Delta Import, Delta Sync, Export), and a status.

A healthy sync shows success on all three operations for both connectors. Look for rows with status stopped-error, stopped-server-down, or completed-export-errors.

⚠️ completed-export-errors is the most common failure state. It means the sync engine processed data but Entra rejected one or more objects during export. Click the row, then click the number in the "Export Errors" column to see specific objects.
✓
04 Identify the specific export error ›

Click the failed export row, then click the error count. The detail pane shows the Error Name, Error Type, and the Distinguished Name of the failing object.

Error NameCommon CauseResolution
InvalidSoftMatchAn AD user's proxyAddress or UPN matches an existing cloud-only userDelete or rename the conflicting cloud user, or hard-match using ImmutableID
AttributeValueMustBeUniqueDuplicate proxyAddress, UPN, or mailNickname across two AD usersFind and fix the duplicate attribute in AD, then re-sync
DataValidationFailedAttribute value exceeds max length or contains invalid charactersFix the attribute value in AD (often a phone number or address field)
LargeObjectUser has too many group memberships (>500 direct) or proxy addressesReduce direct group memberships or use nested groups
IdentityDataValidationFailedUPN contains invalid characters or uses an unverified domainFix the UPN in AD to use a verified domain suffix
✓
05 Inspect the failing object in the connector space ›

In Synchronization Service Manager, go to the Connectors tab. Right-click the ssolabs1001.onmicrosoft.com connector → Search Connector Space. Search by Distinguished Name from the error. Click the object, then click Properties to see its pending export attributes.

Compare the pending export values against what Entra expects. Common mismatches include a UPN that doesn't match an Entra verified domain, or a proxyAddress collision with another synced object.

ℹ️ You can also search by Metaverse Distinguished Name on the AD connector to see what the source data looks like before sync rules transform it.
✓
06 Fix the source data in Active Directory ›

Based on the error from Step 4, make the necessary correction directly in Active Directory. Common fixes:

PowerShell — fix duplicate UPN # Find users with duplicate UPN prefix
Get-ADUser -Filter "UserPrincipalName -like 'duplicate.user*'" -Properties UserPrincipalName

# Fix the UPN
Set-ADUser -Identity "CN=Duplicate User,OU=..." -UserPrincipalName "corrected.user@yearwood.local"
PowerShell — fix duplicate proxyAddress # Find all users with a specific proxy address
Get-ADUser -Filter * -Properties proxyAddresses | Where-Object { $_.proxyAddresses -match "duplicate@" }
✓
07 Force a delta sync cycle ›

After fixing the source data, trigger an immediate delta sync instead of waiting for the next 30-minute automatic cycle.

PowerShell 5.1 Start-ADSyncSyncCycle -PolicyType Delta

Watch the Operations tab in Synchronization Service Manager. The delta import, delta sync, and export should all complete with success status. If the same export error persists, the fix in Step 6 didn't resolve the root cause — go back and re-inspect.

⚠️ Only use -PolicyType Initial (full sync) as a last resort. Full syncs re-process every object in the directory and can take hours on large tenants. Delta sync only processes objects that changed since the last run.
✓
08 Verify the object now exists in Entra ›

After the delta sync completes successfully, verify the previously failing object now appears in the Entra portal under Users (or Groups). Confirm On-premises synced: Yes is displayed on the user profile.

You can also verify programmatically:

PowerShell — Graph Get-MgUser -Filter "userPrincipalName eq 'corrected.user@ssolabs1001.onmicrosoft.com'" | Select DisplayName, OnPremisesSyncEnabled
✓
09 Document and close ›

Record the following in the incident ticket:

Root cause: The specific error name and which AD attribute caused it.
Resolution: What was changed in AD and when the delta sync confirmed the fix.
Prevention: Whether an AD attribute validation policy should be implemented to prevent the same type of data from entering AD in the future.

ℹ️ Export the Entra Audit Log filtered to "Azure AD Connect" source for the last 24 hours as evidence. This shows every sync operation and its outcome.
RB-002
Conditional Access Blocking Legitimate Users
A user reports they cannot sign in to a cloud app. The sign-in is being blocked or challenged by a Conditional Access policy when it should be allowed. Common after CA policy changes, new app registrations, or device compliance state changes.
P1 — High Conditional Access ⏱ Est. 10–30 min
Symptoms & Triggers
🚫 User sees "You cannot access this right now" or AADSTS53003 error
🔄 User is stuck in an MFA loop — completes MFA but gets challenged again
📱 User can sign in from their phone but not their laptop (or vice versa)
⏰ Started happening after a CA policy was created or modified
Error Code Reference
Error CodeNameMeaning
53003BlockedByConditionalAccessAccess blocked by a CA policy. The grant controls were not satisfied.
50076UserStrongAuthEnrollmentRequiredUser needs to register for MFA but hasn't completed registration.
50097DeviceAuthenticationRequiredCA policy requires a compliant or Entra-joined device but the device doesn't meet requirements.
53000DeviceNotCompliantIntune compliance policy marks the device as non-compliant.
530003BlockedBySecurityDefaultsSecurity Defaults (not CA) is blocking the sign-in. Usually means Security Defaults wasn't disabled.
Diagnostic & Resolution Steps
✓
01 Pull the sign-in log entry for the failed attempt ›

In the Entra portal go to Identity → Monitoring → Sign-in logs. Filter by the affected user's UPN and the time range of the reported failure. Click the failed sign-in entry (Status: Failure or Interrupted).

On the Basic info tab, record the Sign-in error code and Failure reason. Then click the Conditional Access tab — this shows exactly which policies were evaluated, which ones applied, and which grant controls succeeded or failed.

ℹ️ The Conditional Access tab on the sign-in log is the single most important diagnostic view. It lists every CA policy with its evaluation result: Success (policy applied, controls satisfied), Failure (policy applied, controls NOT satisfied — this is your blocker), or Not applied (policy conditions didn't match this sign-in).
✓
02 Identify which CA policy is blocking ›

In the Conditional Access tab, find the policy with result Failure. Note the policy name and which grant control failed. Common scenarios:

Which grant control failed?
Require MFA → User hasn't registered for MFA or MFA challenge failed. Go to Step 3.
Compliant device → Device is not Intune-enrolled or compliance policy marks it non-compliant. Go to Step 5.
Block access → User matched a blocking policy (e.g., CA-003 legacy auth). Go to Step 6.
✓
03 Simulate with What If to confirm the policy logic ›

Go to Conditional Access → Policies → What If. Enter the affected user, the target app, and the sign-in conditions (device platform, client app) that match their failed attempt. Click What If.

Compare the What If result against the sign-in log. They should match — the same policies should fire with the same grant controls. If they don't match, the user's actual sign-in conditions may differ from what they reported (e.g., they said "browser" but are using an Outlook desktop client).

✓
04 Check if the user is incorrectly included or excluded ›

Open the blocking CA policy. Check the Users assignment:

• Is the user in the Include scope when they shouldn't be?
• Should the user be in an Exclude group but isn't?
• Is the user's group membership stale (they changed departments but the group wasn't updated)?

Also check the Target resources — does the policy target "All cloud apps" when it should only target specific apps?

⚠️ Never resolve a CA block by adding the user to an exclusion group as a permanent fix. That's a security bypass. Instead, fix the root cause — either the policy scope is too broad, or the user needs to satisfy the grant control (register for MFA, enroll their device, etc.).
✓
05 Resolve device compliance issues (if applicable) ›

If error code 50097 or 53000, check the device's compliance state in Intune (Devices → All devices, search by user). Common issues:

• Device not enrolled in Intune
• OS version out of compliance (needs update)
• BitLocker not enabled
• Antivirus definitions outdated

The user may need to re-enroll their device or update their OS to satisfy the compliance policy.

✓
06 Check for legacy auth blocking (CA-003 pattern) ›

If the sign-in log shows the Client app as "Exchange ActiveSync", "Other clients", "IMAP", or "SMTP" — the user is using a legacy authentication protocol that CA-003 is correctly blocking.

This is working as designed. The resolution is to migrate the user to a modern auth client (Outlook desktop, Outlook mobile, or webmail), not to weaken the CA policy.

🚨 Never disable or create exceptions in your legacy auth blocking policy. Legacy protocols cannot perform MFA, so any exception permanently bypasses your MFA enforcement for that user. If a user "needs" legacy auth, escalate to security for risk acceptance.
✓
07 Apply the fix and verify ›

After identifying and fixing the root cause (policy scope correction, user MFA registration, device enrollment), ask the user to attempt sign-in again. Check the sign-in logs to confirm the new attempt shows Status: Success with the CA policy result showing Success (controls satisfied).

✓
08 Document and close ›

Root cause: Which CA policy blocked the user, which grant control failed, and why the user's sign-in conditions triggered the policy.
Resolution: What was changed — policy scope adjusted, user registered for MFA, device enrolled, or user migrated off legacy client.
Prevention: Whether the CA policy scope needs refinement or whether onboarding procedures should include MFA registration and device enrollment before the user's first day.

RB-003
SAML SSO Assertion Failure
A user authenticates successfully with Entra but the service provider (SP) rejects the SAML assertion. The user sees an error page on the application side after completing Entra sign-in. Common after certificate rotation, app re-registration, or claim mapping changes.
P2 — Medium SSO / SAML 2.0 ⏱ Est. 15–30 min
Symptoms & Triggers
🔴 User completes Entra login but lands on the app's error page
🔴 App logs show "Invalid SAML response" or "Signature validation failed"
🔴 Sign-in log shows Success on Entra side but user can't access the app
🔴 Recently rotated the SAML signing certificate or changed claim mappings
Diagnostic & Resolution Steps
✓
01 Check the Entra sign-in log SAML tab ›

Go to Sign-in logs, find the user's sign-in attempt for the SAML app. Even if Entra shows Success, click the entry and look at the Additional Details tab — it may contain SAML-specific warnings.

If available, check the SAML token tab. This shows the assertion Entra generated, including the NameID, attributes, and audience. This is the same data you'd see by decoding the base64 SAMLResponse from the browser.

✓
02 Capture the SAMLResponse from the browser ›

Open browser DevTools (F12) → Network tab → enable Preserve log. Reproduce the sign-in flow. Look for the POST to the app's ACS URL. In the Payload tab, copy the SAMLResponse value.

Decode it with a base64 decoder or use a SAML decoder tool. Inspect these fields:

Key fields to verify Destination → Must match the app's ACS URL exactly
Audience → Must match the app's Entity ID / Identifier
NameID → Must be in the format the app expects (email, UPN, persistent)
NotOnOrAfter → Must not be in the past (check for clock skew)
Signature → Certificate must match what the app has configured
✓
03 Verify the ACS URL and Entity ID match ›

The most common SAML failure is a URL mismatch. Compare three values:

1. The Destination in the SAML assertion (what Entra sent)
2. The Reply URL / ACS URL configured in the Entra enterprise app
3. The ACS URL configured on the service provider side

All three must be identical — including protocol (http vs https), port number, and trailing slash.

⚠️ A trailing slash mismatch (https://app.com/sso/acs vs https://app.com/sso/acs/) will cause the SP to reject the assertion even though the URLs look "the same" to a human.
✓
04 Check the signing certificate expiration ›

In the Entra portal, go to the enterprise app → Single sign-on → SAML Signing Certificate section. Check the Expiration Date. If expired, the SP will reject every assertion because the signature can't be validated.

If the certificate was recently rotated:

1. Download the new certificate (Base64 format) from Entra
2. Upload it to the SP's SAML configuration
3. Test sign-in again

🚨 Certificate rotation requires updating BOTH sides. If you rotate the cert in Entra but don't upload the new cert to the SP, every SAML sign-in for that app will break immediately. Always coordinate cert rotation with the app owner.
✓
05 Verify the NameID format and value ›

Check what NameID format the SP expects vs what Entra is sending. In the decoded assertion, look at the NameID Format attribute:

FormatTypical ValueWhen Used
emailAddressuser@domain.comMost web apps — matches user by email
persistentOpaque GUIDPrivacy-preserving — app can't derive the email
unspecifiedVariesApp accepts any format — Entra decides

If the app expects emailAddress but Entra is sending persistent (or vice versa), reconfigure the NameID format in the enterprise app's SAML settings under User Attributes & Claims.

✓
06 Check claim mappings for required attributes ›

Some SPs require specific attributes in the assertion beyond NameID — for example, a groups claim, a role claim, or a department attribute. If a required claim is missing or has the wrong schema URI, the SP will reject the assertion.

Compare the AttributeStatement section of the decoded assertion against the SP's documentation for required claims. Add any missing claims in the Entra enterprise app under User Attributes & Claims.

✓
07 Check for clock skew ›

The SAML assertion has a validity window defined by NotBefore and NotOnOrAfter timestamps. If the SP's server clock is significantly off from Entra's clock, the SP may reject the assertion as expired or not-yet-valid.

Entra typically allows a 5-minute skew window (NotBefore is set 5 minutes before IssueInstant). If the SP's clock is off by more than 5 minutes, you'll need to fix the time synchronization on the SP server — the issue is on the SP side, not the IdP side.

✓
08 Test and verify the fix ›

After applying the fix (URL correction, cert upload, claim mapping change), use the Test button on the Entra enterprise app's SAML SSO page. Entra will walk through the sign-in flow and show a detailed result including any remaining issues.

Also test with the actual affected user — have them clear their browser cache and attempt sign-in again. Confirm success in both the Entra sign-in log and the SP's application logs.

✓
09 Document and close ›

Root cause: Which specific mismatch caused the SP to reject the assertion — URL, certificate, NameID format, missing claim, or clock skew.
Resolution: What was corrected and on which side (IdP or SP).
Prevention: Set a calendar reminder for the next certificate rotation (typically 3 years). Add the SAML signing certificate expiration date to your monitoring system.

RB-004
MFA Registration Lockout
A user cannot complete MFA because they lost their phone, switched devices without transferring their authenticator, or their MFA registration became corrupted. They are locked out of all apps that require MFA.
P1 — High Authentication ⏱ Est. 5–15 min
Symptoms & Triggers
📱 User reports lost or stolen phone — can't complete Authenticator push
🔄 User got a new phone but didn't transfer Authenticator app
❌ MFA prompt says "Something went wrong" or "Your request was denied"
🆕 New hire was never given the opportunity to register MFA methods
Diagnostic & Resolution Steps
✓
01 Verify the user's identity through an out-of-band channel ›

This is the most critical step in the entire runbook. Before resetting anyone's MFA, you must verify they are who they claim to be. MFA reset is the #1 social engineering attack vector — an attacker calls the helpdesk, claims they lost their phone, and gets MFA reset on a target account.

Verify identity using a method that is not the same method the user is claiming to have lost:

• Video call and verify against their badge photo or HR record
• Manager confirmation via a separate communication channel
• In-person verification at the IT desk
• Pre-established security questions (weakest option)

🚨 Never reset MFA based solely on an email or chat message. If the attacker has compromised the user's password, they may also have access to their email. The verification must use a channel the attacker cannot control.
✓
02 Check the user's current MFA registration state ›

In the Entra portal, go to Users → select the user → Authentication methods. This page shows all registered MFA methods: Authenticator app, phone numbers, FIDO2 keys, etc.

Determine what methods exist and which ones the user still has access to. If they have a backup method (e.g., a phone number for SMS), they may be able to use that instead of a full reset.

PowerShell — Graph Get-MgUserAuthenticationMethod -UserId "elena.vasquez@ssolabs1001.onmicrosoft.com"
✓
03 Determine the correct remediation path ›
Does the user have any working MFA methods left?
YES → Have them sign in using the backup method, then add their new device as an additional method from their Security Info page. Skip to Step 7.
NO → All methods are inaccessible. Proceed to Step 4 to delete the old methods and require re-registration.
✓
04 Delete the old authentication methods ›

In the Entra portal on the user's Authentication methods page, delete each registered method that the user no longer has access to. This includes the Authenticator app registration, phone numbers for old devices, etc.

Via PowerShell for bulk operations:

PowerShell — Graph # List all methods
$methods = Get-MgUserAuthenticationMethod -UserId "elena.vasquez@ssolabs1001.onmicrosoft.com"

# Delete a specific phone method
Remove-MgUserAuthenticationPhoneMethod -UserId "elena.vasquez@..." -PhoneAuthenticationMethodId $methodId

# Delete Microsoft Authenticator registration
Remove-MgUserAuthenticationMicrosoftAuthenticatorMethod -UserId "elena.vasquez@..." -MicrosoftAuthenticatorAuthenticationMethodId $methodId
✓
05 Issue a Temporary Access Pass (TAP) ›

A Temporary Access Pass is a time-limited passcode that satisfies MFA requirements, allowing the user to sign in and register new MFA methods. This is the recommended approach over "require re-registration" because it provides a controlled, auditable access window.

In the Entra portal on the user's Authentication methods page, click + Add authentication method → Temporary Access Pass. Set the lifetime (default: 1 hour) and whether it's single-use or multi-use.

ℹ️ TAP requires an Entra ID P1 or P2 license and must be enabled in the Authentication methods policy before it can be issued. If TAP is not available, use Step 6 as the alternative approach.
✓
06 Alternative: require re-registration at next sign-in ›

If TAP is not available, you can require the user to re-register MFA at their next sign-in. After deleting their old methods in Step 4, the next time they sign in and a CA policy requires MFA, Entra will prompt them to set up new methods.

However, this creates a chicken-and-egg problem: the user needs to sign in to register MFA, but CA-001 requires MFA to sign in. To break the cycle, you may need to temporarily add the user to a CA exclusion group, let them sign in and register, then remove them from the exclusion group immediately.

🚨 If you use the temporary exclusion approach, set a calendar reminder to remove the user from the exclusion group within 1 hour. Never leave a user permanently excluded from MFA enforcement.
✓
07 Confirm new MFA methods are registered ›

After the user signs in with the TAP (or via temporary exclusion), they should be redirected to https://aka.ms/mysecurityinfo to register new methods. Confirm they register at least two methods (e.g., Authenticator app + phone number as backup).

Verify in the Entra portal that the new methods appear on their Authentication methods page.

✓
08 Document and close ›

Root cause: How the user lost access to their MFA methods (lost device, new phone, corrupted app).
Identity verification: How the user's identity was verified before the reset — this must be recorded for audit purposes.
Resolution: Which methods were deleted, whether TAP or temporary exclusion was used, and what new methods the user registered.
Prevention: Remind users to register multiple MFA methods and to transfer their Authenticator app before switching devices.

RB-005
AD User Not Appearing in Entra
A user was created in Active Directory but does not appear in Entra ID after the expected sync interval (30 minutes). The user cannot sign in to cloud apps and has no Entra profile. This is the most common Tier 1 identity support ticket in hybrid environments.
P2 — Medium Hybrid Sync ⏱ Est. 10–20 min
Symptoms & Triggers
👤 New hire created in AD but not visible in Entra Users after 30+ minutes
🔍 User can authenticate to on-prem resources but not cloud apps
📧 User has no M365 license because their Entra account doesn't exist yet
🔄 HR/IT created the user in an OU that may not be in the sync scope
Diagnostic & Resolution Steps
✓
01 Confirm the user exists in Active Directory ›

Before troubleshooting sync, confirm the user actually exists in AD and has the expected attributes populated.

PowerShell Get-ADUser -Identity "elena.vasquez" -Properties DistinguishedName, UserPrincipalName, Enabled, MemberOf

# Verify:
# - Enabled = True
# - DistinguishedName shows the correct OU
# - UserPrincipalName is populated
✓
02 Check if the user's OU is in the sync scope ›

This is the #1 cause of "user not syncing." Entra Connect is configured with OU scoping filters — only objects in selected OUs are synced. If the user was created in an OU that's not in scope, they will never appear in Entra.

Check the user's DistinguishedName from Step 1. Compare the OU path against the Entra Connect OU scoping configuration.

Is the user's OU in the sync scope?
YES → OU is in scope. Proceed to Step 3 to check the connector space.
NO → The OU is not in scope. Either move the user to a scoped OU, or add the OU to the sync scope (Step 3a).

To check OU scoping: Run the Entra Connect configuration wizard → Customize synchronization options → advance to the Domain/OU Filtering page. Checked OUs are in scope.

⚠️ If you add a new OU to the sync scope via the wizard, you must run a full sync afterward (Start-ADSyncSyncCycle -PolicyType Initial) for all objects in that OU to be picked up. A delta sync only catches changes — it won't discover objects in a newly added OU.
✓
03 Search the connector space for the user object ›

Open Synchronization Service Manager → Connectors tab. Right-click the yearwood.local connector → Search Connector Space. Search by Distinguished Name or display name.

Does the object appear in the AD connector space?
YES → The import picked it up. Check if it also exists in the Entra connector space (search the ssolabs1001.onmicrosoft.com connector). If it's in AD connector but NOT in Entra connector, there may be a sync rule filtering it out. Proceed to Step 4.
NO → The import hasn't picked it up yet. Trigger a delta import: Start-ADSyncSyncCycle -PolicyType Delta and wait for it to complete, then search again.
✓
04 Check for sync rule filtering or attribute issues ›

If the object is in the AD connector space but not syncing to Entra, the sync rules may be filtering it out. Common reasons:

• adminDescription contains "Device" — Entra Connect's default rules skip objects with this attribute
• isCriticalSystemObject = True — built-in AD objects are excluded
• sAMAccountName starts with $ — service accounts and computer objects
• userAccountControl includes ACCOUNTDISABLE — some custom sync rules exclude disabled accounts
• Custom sync rule with a scoping filter — check the Synchronization Rules Editor for any custom inbound rules with scoping conditions

Click the object in the AD connector space → Lineage tab to see which sync rules applied and whether any filtered it out.

✓
05 Force a delta sync and monitor ›

After resolving the issue (moved user to correct OU, added OU to scope, fixed filtering attribute), trigger a delta sync:

PowerShell 5.1 Start-ADSyncSyncCycle -PolicyType Delta

Watch the Operations tab in Synchronization Service Manager. You should see:

1. Delta Import on yearwood.local connector — picks up the new/moved user
2. Delta Synchronization — processes the user through sync rules
3. Export on ssolabs1001.onmicrosoft.com connector — pushes the user to Entra

All three should complete with success status. The Export step should show 1 or more adds.

✓
06 Verify the user appears in Entra ›

After the export completes, check the Entra portal → Users → search for the user. Confirm:

• The user appears with On-premises synced: Yes
• The UPN has been rewritten to @ssolabs1001.onmicrosoft.com (or verified custom domain)
• The display name, department, and other attributes match AD

PowerShell — Graph Get-MgUser -Filter "displayName eq 'Elena Vasquez'" | Select DisplayName, UserPrincipalName, OnPremisesSyncEnabled
✓
07 Test cloud app sign-in ›

Have the user attempt to sign in to a cloud app (or test via the My Apps portal at https://myapps.microsoft.com). If the user's account was just synced, remember they will need to register for MFA on first sign-in (due to CA-001). Make sure they have their phone ready for Authenticator setup.

✓
08 Document and close ›

Root cause: Why the user wasn't syncing — OU not in scope, sync rule filtering, attribute issue, or simply waiting for the next delta cycle.
Resolution: What was changed and what sync operation confirmed the fix.
Prevention: Ensure the user provisioning process (JML script or manual creation) always places users in OUs that are within the Entra Connect sync scope. Consider documenting the OU-to-sync-scope mapping so IT staff creating users know which OUs are valid.
